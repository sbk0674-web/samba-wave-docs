왜 확장앱을 쓸까?

더망고와 오버링크, 두 솔루션이 브라우저 확장앱을 사용하는 이유와
서로 다른 서버 운영 방식이 안정성에 미치는 영향을 비교합니다

CHAPTER 01

먼저 쉬운 비유로 이해하기

인터넷 쇼핑몰에서 상품 정보를 가져오는 걸, 도서관에서 책 내용을 베끼는 것에 비유해볼게요.

🤖

확장앱 없이 (서버가 직접)

로봇을 도서관에 보내서 책을 읽게 하는 것과 같아요. 그런데 도서관 입구에서 "로봇은 출입금지!"라는 표지판을 보고 쫓겨납니다. 도서관 회원증(로그인)도 없고, 사서가 로봇을 의심해서 책을 안 보여줘요.

👤

확장앱으로 (사람이 직접)

내가 직접 도서관에 가서 회원증을 보여주고, 책을 펼쳐 읽으면서 중요한 내용만 메모하는 거예요. 사서는 아무런 의심 없이 책을 보여주고, 나는 편하게 메모를 가져옵니다.

👓

확장앱 = "특수 안경"

브라우저 확장앱은 특수 안경 같은 거예요. 내가 평소처럼 쇼핑몰을 구경하면, 안경이 자동으로 상품 정보를 읽어서 메모장(서버)에 적어줘요. 쇼핑몰 입장에서는 그냥 평범한 손님이 구경하는 것처럼 보입니다.

📋

왜 둘 다 확장앱을 쓸까?

쇼핑몰들은 로봇(서버)이 데이터를 긁어가는 걸 적극적으로 차단해요. CAPTCHA, 로그인 강제, IP 차단 등... 하지만 실제 사람(브라우저 확장앱)은 차단할 수 없어요. 그래서 더망고와 오버링크 모두 확장앱을 사용합니다.

CHAPTER 02

확장앱을 쓰는 진짜 이유 5가지

더망고와 오버링크 모두 확장앱을 사용하는 건 같습니다. 서버가 직접 사이트에 접근하면 생기는 문제들을 확장앱이 해결해주기 때문이에요.

문제 서버가 직접 접근하면 확장앱을 쓰면
봇 감지 "로봇이 접속 중입니다" 차단 차단됨 실제 사람이 보는 브라우저 통과
로그인 CAPTCHA, 2단계 인증 등 자동 로그인 거의 불가능 이미 로그인된 브라우저 세션 그대로 사용
쿠키/세션 서버가 직접 관리해야 함 (만료, 갱신, 암호화 복호화...) 브라우저가 알아서 관리해줌
자바스크립트 SPA 사이트는 Puppeteer로 무거운 브라우저를 돌려야 함 브라우저가 이미 다 실행해놓은 완성된 화면을 읽음
개발 난이도 사이트마다 1~2주 개발, 봇 우회 기법 연구 매우 어려움 개발자도구에서 셀렉터만 찾으면 끝 1~2시간
CHAPTER 03

그럼 더망고와 오버링크는 뭐가 다를까?

둘 다 확장앱을 쓰고, 둘 다 확장앱 리로드가 필요 없는 건 같아요. 차이는 서버를 어떻게 운영하느냐에 있습니다.

더망고 2년간 끊김 0회
내가 쇼핑몰 구경
>
확장앱이 DOM 읽기
>
서버에 POST 전송
핵심: 확장앱은 "보고 보내기"만 한다. 서버 코드가 바뀌어도 확장앱에는 영향이 없다. 서버가 항상 켜져있는 호스팅(mycafe24)이라 배포 개념이 없다.
오버링크 배포 시 서비스 중단
내가 쇼핑몰 구경
>
확장앱이 DOM 읽기
>
서버 배포 중... 꺼짐
핵심: 확장앱 동작은 같지만, 서버가 자체 운영 서버라서 코드 업데이트(배포) = 서버 재시작 = 서비스 중단. 배포를 자주 할수록 사용자에게 끊김이 잦아진다.
더망고 오버링크
확장앱 역할 DOM 읽기 + 서버 전송 DOM 읽기 + 서버 전송
확장앱 리로드 필요 없음 필요 없음
서버 호스팅 mycafe24 (관리형 호스팅) 항상 켜짐 자체 서버 (직접 관리) 배포 시 꺼짐
서버 배포 배포 개념 없음 (코드가 호스팅에서 바로 반영) 코드 수정 → 서버 재시작 → 서비스 중단
2년간 끊김 0회 수시로
안정성 비결 서버가 죽을 일이 없는 구조 서버 배포 전략(무중단 등)이 필요
CHAPTER 04

왜 오버링크는 배포하면 끊길까?

"배포"라는 건 서버의 코드를 새 버전으로 바꾸는 작업이에요. 이걸 식당에 비유하면 이해가 쉬워요.

🏪

더망고 = 프랜차이즈 식당

본사(mycafe24)가 운영하는 식당이에요. 메뉴를 바꿔도 주방이 멈추지 않아요. 메뉴판만 슬쩍 바꾸면 끝. 손님(확장앱)은 항상 주문할 수 있어요.

🍳

오버링크 = 1인 식당

사장님이 직접 요리도 하고 서빙도 하는 식당이에요. 메뉴를 바꾸려면 가게 문을 닫고 주방을 정비해야 해요. 그 동안 손님(확장앱)은 "문 닫았네..." 하고 돌아가야 합니다.

💡

해결 방법: 무중단 배포

오버링크도 끊김 없이 배포할 수 있어요. 핵심은 "교대 근무"입니다. 새 버전 서버를 먼저 켜놓고, 준비가 되면 손님을 새 서버로 안내하고, 옛 서버를 끄는 거예요. 이걸 IT에서는 "블루-그린 배포" 또는 "롤링 배포"라고 불러요. 하지만 구현이 복잡하고, 소규모 서비스에서는 "잠깐 끄고 켜는 게 더 빠르니까..." 하고 넘어가는 경우가 많습니다.

왜 이렇게 차이가 날까?

확장앱은 "이미 완성된 화면을 보고 메모"하는 거예요. 개발자도구(F12)에서 원하는 부분의 이름(셀렉터)만 찾으면 끝이에요.

서버에서 직접 수집하면 "빈 종이부터 그림을 다시 그려야" 해요. 봇 감지 우회, 로그인 자동화, 쿠키 암호화 복호화, JavaScript 실행 환경 구축... 같은 데이터를 얻는데 10~50배 더 어렵습니다.

SUMMARY

정리

확장앱을 쓰는 이유

  • 쇼핑몰의 봇 차단을 피할 수 있다 (실제 사람의 브라우저이니까)
  • 이미 로그인된 상태를 그대로 쓸 수 있다 (로그인 자동화 불필요)
  • 브라우저가 다 그려놓은 완성된 화면을 읽으면 된다 (JS 렌더링 불필요)
  • 개발이 10배 이상 쉽고, 유지보수도 간단하다

더망고 vs 오버링크 차이

  • 둘 다 확장앱을 사용하고, 둘 다 확장앱 리로드가 필요 없는 건 같다
  • 차이는 "서버 운영 방식"에 있다
  • 더망고: 관리형 호스팅 → 배포 개념 자체가 없다 → 2년간 끊김 0회
  • 오버링크: 자체 서버 → 코드 수정할 때마다 배포(재시작) → 서비스 중단 발생
  • 오버링크도 "무중단 배포"를 도입하면 해결 가능하지만, 추가 구현이 필요하다
