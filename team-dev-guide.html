이것부터 이해하기

코드를 쓰기 전에, 우리 프로젝트가 어떤 구조인지 먼저 이해합시다. 여기서 비유를 통해 설명하니 끝까지 읽어주세요.

🎯 우리가 만드는 것
💡 비유로 이해하기

우리 시스템은 "번역기"와 비슷합니다.

소싱처(무신사, 올리브영 등)에서 상품 정보를 가져와서,
판매처(쿠팡, 스마트스토어 등)의 형식에 맞게 변환해서 올려주는 겁니다.

예를 들어, 무신사에서 "나이키 에어맥스"를 가져와서 → 쿠팡 형식으로 바꿔서 → 쿠팡에 등록.
이 과정을 자동으로 해주는 게 우리 시스템입니다.

🧩 플러그인이 뭔가요?
💡 비유로 이해하기

플러그인은 "USB 장치"와 같습니다.

컴퓨터 본체(우리 시스템)는 이미 완성되어 있고,
마우스(쿠팡), 키보드(스마트스토어), 웹캠(11번가)처럼
USB만 꽂으면 바로 인식되죠? 드라이버 설치 같은 거 필요 없이요.

우리 플러그인도 마찬가지입니다. 파일 하나 만들면 자동으로 인식됩니다.
다른 파일을 수정할 필요가 전혀 없어요.

🏪 판매처(마켓) 플러그인

쿠팡, 스마트스토어, 11번가 같은 곳에 상품을 등록하는 코드

역할: 상품 데이터를 해당 마켓 형식으로 변환하고 전송

📦 소싱처 플러그인

무신사, 올리브영 같은 곳에서 상품을 가져오는 코드

역할: 상품 검색, 상세 조회, 가격/재고 갱신

👥 왜 10명이 동시에 해도 괜찮은가?
💡 비유로 이해하기

아파트를 생각해보세요. 10명이 각자 다른 호수에서 인테리어 공사를 합니다.

101호에서 벽지를 바꾸든, 102호에서 바닥을 교체하든 서로 영향이 없죠?
하지만 복도(공용부분)를 동시에 공사하면 부딪히겠죠.

우리 프로젝트도 같습니다:
• 각자 만드는 플러그인 파일 = 각자의 호수 (충돌 없음)
• 공유 파일(__init__.py, base.py) = 복도 (건드리면 안 됨)

용어 사전

이 문서에서 계속 나오는 단어들입니다. 모르면 여기로 돌아오세요.

용어 초등학생한테 설명한다면 예시
Git "되돌리기"가 무한대인 저장 시스템. 코드의 모든 변경 이력을 기록해줌 워드 문서의 "변경 내용 추적"
Repository(저장소) 프로젝트 코드가 들어있는 폴더. 줄여서 "레포" samba-wave 폴더 전체
Branch(브랜치) 원본을 복사해서 따로 작업하는 공간. "평행 세계" 같은 것 feature/tmon-plugin
main 브랜치 "진짜 원본". 여기를 직접 수정하면 안 됨 출판된 책의 원고
Commit(커밋) "저장" 버튼. 지금까지의 변경을 기록으로 남김 게임에서 "세이브" 하는 것
Push(푸시) 내 컴퓨터의 저장 내용을 GitHub(인터넷)에 올리기 구글 드라이브에 업로드
Pull(풀) GitHub(인터넷)의 최신 내용을 내 컴퓨터로 가져오기 구글 드라이브에서 다운로드
Merge(머지) 두 개의 브랜치를 합치기 복사본의 변경을 원본에 반영
PR (Pull Request) "제 코드 좀 확인해주세요" 하고 팀장에게 보내는 요청 과제물 제출 후 선생님 검사
충돌(Conflict) 두 사람이 같은 줄을 동시에 수정해서 Git이 어떤 걸 써야 할지 모르는 상태 같은 문장을 두 사람이 다르게 고침
Plugin(플러그인) 본체를 수정하지 않고 기능을 추가하는 부품 USB 장치
Proxy(프록시) 외부 사이트의 API를 대신 호출해주는 중간 역할 대리인, 통역사
API 프로그램끼리 대화하는 규칙. "이렇게 보내면 이렇게 답해줄게" 자판기의 버튼과 음료 나오는 구멍

개발 환경 세팅

한 번만 하면 되는 초기 세팅입니다. 순서대로 하나씩 따라하세요.

ℹ️
사전 설치 필요 아래 프로그램이 설치되어 있어야 합니다. 없으면 팀장에게 알려주세요.
• Git — git-scm.com 에서 다운로드. 설치 시 기본값 그대로 "Next" 연타하면 됨
• Python 3.12 — python.org 에서 다운로드. 설치 시 "Add Python to PATH" 반드시 체크!
• uv (파이썬 패키지 관리자) — pip보다 10배 빠른 패키지 설치 도구. Python 설치 후 터미널에서 아래 명령어 실행:
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex" 설치 후 터미널을 껐다 다시 열고, uv --version 입력해서 버전이 나오면 성공
• Antigravity (코드 에디터) — 팀장이 제공하는 설치 파일로 설치

GitHub에서 코드 가져오기 (클론)

# 1. 원하는 폴더로 이동 (예: workspace 폴더) cd ~/workspace # 2. GitHub에서 코드 다운로드 git clone https://github.com/sbk0674-web/samba-wave.git # 3. 프로젝트 폴더로 이동 cd samba-wave
💡
Git Bash 여는 법 Windows에서 바탕화면 우클릭 → "Git Bash Here" 클릭.
또는 Antigravity에서 터미널(Ctrl+`) 열기 → bash 선택.

백엔드 환경 만들기

파이썬 가상환경을 만들고, 필요한 패키지를 설치합니다.

터미널 (Git Bash)
# 1. backend 폴더로 이동 cd backend # 2. 파이썬 가상환경 만들기 (가상환경 = 이 프로젝트 전용 파이썬) uv venv # 3. 가상환경 켜기 (Windows) source .venv/Scripts/activate # 4. 필요한 패키지 설치 uv pip install -e . uv pip install -e .[dev]

안 보이면 source .venv/Scripts/activate를 다시 입력하세요.

.env 파일 설정

데이터베이스 비밀번호 등 비밀 정보가 담긴 파일입니다. 아래 순서대로 하세요.

A

예시 파일 복사하기

B

.env 파일 열어서 값 채우기

Antigravity에서 backend/.env 파일을 열고, 팀장이 공유한 값으로 채웁니다.

C

어떤 값을 넣어야 하는지 모르겠다면?

팀장에게 "DB 접속 정보랑 JWT 키 알려주세요" 라고 요청하세요.

backend/.env 파일 내용 설명
# ============================================ # 환경 설정 — 이건 그대로 두세요 # ============================================ environment=development # ============================================ # [필수] 데이터베이스 — 쓰기용 (팀장이 알려줌) # ============================================ WRITE_DB_USER=postgres # DB 접속 아이디 WRITE_DB_PASSWORD=팀장이_알려준_비밀번호 # DB 접속 비밀번호 WRITE_DB_HOST=팀장이_알려준_주소 # DB 서버 주소 WRITE_DB_PORT=5432 # 포트 (보통 5432) WRITE_DB_NAME=railway # DB 이름 # ============================================ # [필수] 데이터베이스 — 읽기용 (쓰기용과 동일하게!) # ============================================ READ_DB_USER=postgres READ_DB_PASSWORD=팀장이_알려준_비밀번호 # ↑ 쓰기용과 같은 값 READ_DB_HOST=팀장이_알려준_주소 # ↑ 쓰기용과 같은 값 READ_DB_PORT=5432 READ_DB_NAME=railway # 개발환경에서는 SSL 끄기 — 이건 그대로 두세요 DB_SSL_REQUIRED=false # ============================================ # [필수] JWT 시크릿 키 (팀장이 알려줌) # ============================================ JWT_SECRET_KEY=팀장이_알려준_시크릿키 # ============================================ # 아래는 전부 선택사항 — 없어도 서버 실행됨! # 필요할 때 팀장이 알려줍니다 # ============================================ # AWS_ACCESS_KEY_ID= # AWS_SECRET_ACCESS_KEY= # ANTHROPIC_API_KEY= # REDIS_URL=
📌
정리: 팀장에게 받아야 할 값 3개
1 DB 비밀번호 (WRITE_DB_PASSWORD)
2 DB 서버 주소 (WRITE_DB_HOST)
3 JWT 시크릿 키 (JWT_SECRET_KEY)

나머지는 위 예시 그대로 두면 됩니다.
🚨
.env 파일은 절대 GitHub에 올리면 안 됩니다! 비밀번호가 전세계에 공개됩니다. 이미 .gitignore에 등록되어 있지만, 혹시라도 직접 추가하지 마세요.

Git이 뭔가요?

💡 비유로 이해하기 — "Google Docs"

Google Docs를 생각해보세요. 여러 사람이 동시에 문서를 편집할 수 있고,
"버전 기록"에서 누가 뭘 바꿨는지 볼 수 있죠?

Git도 똑같습니다. 다만 코드 파일 전용이고, 더 정밀합니다.

Google Docs와 다른 점 하나:
Google Docs는 실시간으로 합쳐지지만,
Git은 "내가 원할 때" 합칩니다. 이게 브랜치라는 개념입니다.

🌿 브랜치 = 평행 세계

브랜치를 만드는 건 "평행 세계를 하나 만든다"고 생각하세요.

브랜치 개념도
┌── feature/tmon (A가 작업 중) ──── 커밋 ── 커밋 ── merge! │ ↓ main ────●────────────────────────────────────────────────────●────▶ │ ↑ └── feature/oliveyoung (B가 작업 중) ─ 커밋 ─ merge! ● = 시작 지점 ▶ = 현재 진행 중 A와 B는 각자 다른 "세계"에서 작업하다가, 완료되면 main에 합침
  • 1 main 브랜치는 "진짜 세계"입니다. 여기 있는 코드가 실제로 돌아가는 코드예요.
  • 2 feature 브랜치는 "연습 세계"입니다. 여기서 마음껏 실험하고, 완성되면 진짜 세계에 합칩니다.
  • 3 각자 다른 브랜치에서 작업하기 때문에, 다른 사람 작업에 영향을 주지 않습니다.
5

매일 하는 Git 루틴

매일 아침 작업을 시작하기 전에 반드시 하는 과정입니다. 이걸 안 하면 옛날 코드 위에서 작업하게 됩니다.

🌅 아침 — 최신 코드 가져오기
1

main 브랜치로 이동

먼저 "진짜 원본"이 있는 곳으로 갑니다.

터미널
git checkout main
⚠️
에러가 나면? → 아직 저장 안 한 변경이 있다는 뜻입니다.
먼저 git stash를 입력하세요 (임시 보관). 나중에 git stash pop으로 복구.
2

GitHub에서 최신 코드 다운로드

다른 사람이 밤사이에 합친(merge한) 코드를 가져옵니다.

터미널
git pull origin main
3

내 브랜치로 돌아가서 최신 코드 합치기

내 작업 브랜치에도 최신 코드를 반영합니다.

터미널
# 내 브랜치로 돌아가기 git checkout feature/tmon-plugin # main의 최신 코드를 내 브랜치에 합치기 git merge main
✅
이제 내 브랜치에도 다른 사람들의 최신 코드가 반영되었습니다.
안심하고 작업을 시작하세요!
🆕 처음 시작할 때 — 브랜치 만들기

프로젝트에 처음 참여할 때, 또는 새로운 작업을 시작할 때 한 번 합니다.

터미널
# 1. 먼저 main에서 최신 코드 가져오기 git checkout main git pull origin main # 2. 새 브랜치 만들기 (main을 기반으로) git checkout -b feature/내마켓이름-plugin # 예시들: git checkout -b feature/tmon-plugin # 판매처: 티몬 git checkout -b feature/oliveyoung-sourcing # 소싱처: 올리브영 git checkout -b feature/interpark-plugin # 판매처: 인터파크
작업 종류 브랜치 이름 규칙 예시
판매처 feature/마켓이름-plugin feature/tmon-plugin
소싱처 feature/소싱처이름-sourcing feature/oliveyoung-sourcing
버그수정 fix/설명 fix/coupang-auth-error
6

커밋하기 (저장하기)

💡 비유로 이해하기 — "게임 세이브"

RPG 게임에서 보스전 전에 세이브하잖아요?
커밋도 마찬가지입니다. 작업하다가 "여기까지는 잘 됐다!" 하면 커밋(세이브)하세요.
실수해도 이전 세이브 지점으로 돌아갈 수 있습니다.

📝 커밋 3단계
1

변경 사항 확인하기

"내가 뭘 바꿨지?" 확인합니다.

터미널
git status

그러면 이런 결과가 나옵니다:

결과 예시
On branch feature/tmon-plugin Untracked files: ← "새로 만든 파일"이라는 뜻 backend/backend/domain/samba/plugins/markets/tmon.py backend/backend/domain/samba/proxy/tmon.py
2

파일을 스테이징하기 (커밋할 파일 선택)

"이 파일을 커밋에 포함시킬게요"라고 Git에게 알려줍니다.

✅ 올바른 방법 — 파일을 하나씩 지정
git add backend/backend/domain/samba/plugins/markets/tmon.py git add backend/backend/domain/samba/proxy/tmon.py
❌ 틀린 방법 — 전부 다 추가
# 절대 이렇게 하지 마세요!!! git add . git add -A
🚨
왜 git add .이 위험한가요? git add .은 폴더 안의 모든 파일을 추가합니다.
내가 모르는 사이에 .env(비밀번호), 다른 사람 파일, 불필요한 파일이 함께 올라갈 수 있습니다.
반드시 내 파일만 이름을 지정해서 추가하세요.

판매처(마켓) 플러그인 만들기

"티몬" 판매처 플러그인을 예시로, 한 줄 한 줄 설명합니다.

📄 파일 1 — 플러그인 (plugins/markets/tmon.py)
backend/backend/domain/samba/plugins/markets/tmon.py
"""티몬 마켓 플러그인.""" from __future__ import annotations from typing import Any from backend.domain.samba.plugins.market_base import MarketPlugin from backend.utils.logger import logger class TmonPlugin(MarketPlugin): # ────────────────────────────────────────── # 이 3가지가 가장 중요! 다른 마켓과 겹치면 안 됩니다 # ────────────────────────────────────────── market_type = "tmon" # 영문 소문자, 시스템 내부 식별자 policy_key = "티몬" # UI에 표시될 한글 이름 required_fields = ["name", "sale_price"] # 필수 필드 def transform(self, product: dict, category_id: str, **kwargs) -> dict: """상품 데이터 → 티몬 API 형식으로 변환.""" from backend.domain.samba.proxy.tmon import TmonClient return TmonClient.transform_product(product, category_id) async def execute( self, session, product: dict, creds: dict, category_id: str, account, existing_no: str, ) -> dict[str, Any]: """티몬에 상품 등록 또는 수정.""" from backend.domain.samba.proxy.tmon import TmonClient # 인증 정보 꺼내기 (creds는 base가 자동으로 넣어줌) api_key = creds.get("apiKey", "") if not api_key: return {"success": False, "message": "티몬 API Key 없음"} client = TmonClient(api_key) data = self.transform(product, category_id) # existing_no가 있으면 = 이미 등록된 상품 → 수정 # existing_no가 없으면 = 새 상품 → 등록 if existing_no: result = await client.update_product(existing_no, data) return {"success": True, "message": "티몬 수정 성공", "data": result} else: result = await client.register_product(data) return {"success": True, "message": "티몬 등록 성공", "data": result}
🔑
핵심 포인트
  • 1MarketPlugin을 상속(extends) 합니다. 부모가 인증, 에러처리, 정책 주입을 다 해줌
  • 2transform() — 상품 데이터를 마켓 API 형식으로 바꾸는 함수. 반드시 구현
  • 3execute() — 실제 API를 호출하는 함수. 반드시 구현
  • 4creds는 부모 클래스가 자동으로 채워줍니다. API 키 같은 인증정보가 들어있어요
📄 파일 2 — 프록시 클라이언트 (proxy/tmon.py)
backend/backend/domain/samba/proxy/tmon.py
"""티몬 API 클라이언트. 실제 티몬 서버와 HTTP 통신하는 코드. 플러그인(tmon.py)에서 이 클래스를 가져다 씁니다. """ import httpx from backend.utils.logger import logger # 티몬 API의 기본 주소 (티몬 개발자 문서에서 확인) BASE_URL = "https://api.tmon.co.kr" class TmonClient: """티몬 API와 실제로 통신하는 클래스.""" def __init__(self, api_key: str): """API 키를 받아서 저장.""" self.api_key = api_key async def register_product(self, data: dict) -> dict: """상품 신규 등록. data: transform_product()로 변환된 데이터 """ async with httpx.AsyncClient() as client: resp = await client.post( f"{BASE_URL}/v1/products", json=data, headers={"Authorization": f"Bearer {self.api_key}"}, timeout=30, ) resp.raise_for_status() # 에러 발생 시 예외 던짐 return resp.json() async def update_product(self, product_no: str, data: dict) -> dict: """기존 상품 수정.""" async with httpx.AsyncClient() as client: resp = await client.put( f"{BASE_URL}/v1/products/{product_no}", json=data, headers={"Authorization": f"Bearer {self.api_key}"}, timeout=30, ) resp.raise_for_status() return resp.json() @staticmethod def transform_product(product: dict, category_id: str) -> dict: """우리 상품 데이터 → 티몬 API 형식으로 변환. ※ 여기가 가장 중요한 부분! 티몬 API 문서를 보면서 각 필드를 매핑합니다. """ return { "productName": product.get("name", ""), "salePrice": product.get("sale_price", 0), "categoryCode": category_id, "description": product.get("detail_html", ""), # ↑ 마켓 API 문서를 보면서 필드를 하나씩 추가하세요 }
📌
프록시 파일에서 하는 일
  • 1register_product() — 마켓 API에 상품을 새로 등록하는 HTTP 요청
  • 2update_product() — 이미 등록된 상품을 수정하는 HTTP 요청
  • 3transform_product() — 우리 데이터를 마켓이 원하는 형식으로 변환 (가장 중요!)

"""올리브영 소싱처 플러그인.""" import logging from typing import TYPE_CHECKING from backend.domain.samba.plugins.sourcing_base import SourcingPlugin if TYPE_CHECKING: from backend.domain.samba.collector.refresher import RefreshResult logger = logging.getLogger(__name__) class OliveYoungPlugin(SourcingPlugin): """올리브영 소싱처 플러그인.""" # ────────────────────────────────────────── # 이 3가지 설정이 핵심 # ────────────────────────────────────────── site_name = "OLIVEYOUNG" # 대문자! 시스템 내부 식별자 concurrency = 3 # 동시에 보내는 요청 수 (사이트가 느리면 줄이기) request_interval = 0.5 # 요청 사이 대기 시간(초) (차단 방지) async def search(self, keyword: str, **filters) -> list[dict]: """키워드로 상품 검색.""" from backend.domain.samba.proxy.oliveyoung import OliveYoungClient client = OliveYoungClient() return await self.safe_call(client.search_products(keyword)) async def get_detail(self, site_product_id: str) -> dict: """상품 상세 정보 조회.""" from backend.domain.samba.proxy.oliveyoung import OliveYoungClient client = OliveYoungClient() return await self.safe_call(client.get_detail(site_product_id)) async def refresh(self, product) -> "RefreshResult": """가격, 재고 정보 갱신.""" from backend.domain.samba.proxy.oliveyoung import OliveYoungClient from backend.domain.samba.collector.refresher import RefreshResult client = OliveYoungClient() try: detail = await self.safe_call( client.get_detail(product.site_product_id) ) return RefreshResult( product_id=product.id, sale_price=detail.get("sale_price"), is_sold_out=detail.get("is_sold_out", False), ) except Exception as e: return RefreshResult(product_id=product.id, error=str(e))

자주 묻는 질문 (FAQ)

Q. 내가 어떤 브랜치에 있는지 어떻게 알아요?
git branch # 별표(*)가 붙은 게 지금 있는 브랜치 # main # * feature/tmon-plugin ← 여기 있음
Q. 실수로 main에 커밋해버렸어요!

아직 push 안 했다면:

# 커밋을 되돌리기 (코드는 유지, 커밋만 취소) git reset --soft HEAD~1 # 새 브랜치 만들고 거기서 다시 커밋 git checkout -b feature/내마켓-plugin git add 내파일.py git commit -m "메시지"

이미 push 했다면 → 팀장에게 즉시 연락하세요.

Q. git pull 했더니 이상한 에디터가 열렸어요!

merge 커밋 메시지를 작성하라는 뜻입니다.

# vim 에디터가 열렸을 때: # 1. :q 입력하고 Enter (그냥 나가기) # 또는 # 1. Esc 키 누르기 # 2. :wq 입력하고 Enter (저장하고 나가기)
Q. 서버가 안 켜져요! 에러 메시지가 나와요

자주 발생하는 원인들:

  • 1가상환경이 꺼져있음 → source .venv/Scripts/activate 실행
  • 2.env 파일이 없음 → 팀장에게 파일 요청
  • 3import 에러 → 내 플러그인 코드에 오타가 있을 수 있음. 에러 메시지 확인
  • 4DB 연결 실패 → VPN이 꺼져있거나 .env의 DB 주소가 틀린 경우
Q. transform 함수에 어떤 필드를 넣어야 하나요?

해당 마켓의 API 개발 문서를 봐야 합니다.
마켓마다 요구하는 필드가 다릅니다 (상품명, 가격, 카테고리, 배송비, 이미지 등).

기존 구현체를 참고하세요:
• 간단한 예: plugins/markets/elevenst.py (71줄)
• 복잡한 예: plugins/markets/coupang.py (143줄)

Q. 모르겠으면 어떡하나요?

1. 이 가이드를 다시 읽어보기
2. 기존 코드 파일 열어서 참고하기 (복사 → 수정하면 됨!)
3. 팀 채팅방에 질문하기
4. 팀장에게 화면 공유하며 도움 요청
