우리 프로젝트 이해하기

🧩 플러그인 = USB 장치

컴퓨터 본체(우리 시스템)는 이미 완성되어 있고, 마우스(쿠팡), 키보드(스마트스토어)처럼 USB만 꽂으면 바로 인식됩니다.
우리 플러그인도 마찬가지 — 파일 하나 만들면 자동으로 인식됩니다. 다른 파일을 수정할 필요가 없습니다.

🏪 판매처(마켓) 플러그인

쿠팡, 스마트스토어, 11번가 같은 곳에 상품을 등록하는 코드

역할: 상품 데이터를 해당 마켓 형식으로 변환하고 전송

📦 소싱처 플러그인

무신사, 올리브영 같은 곳에서 상품을 가져오는 코드

역할: 상품 검색, 상세 조회, 가격/재고 갱신

🌿 Git과 브랜치 — 왜 충돌이 안 나는가

Git은 코드의 모든 변경 이력을 기록하는 시스템입니다. Google Docs의 "버전 기록"과 비슷하지만, "내가 원할 때" 합치는 게 다릅니다. 이게 브랜치(Branch, 평행 세계) 개념입니다.

브랜치 전략 (프로덕션 기준)
┌── feature/tmon-plugin (팀원 A) ──── 커밋 ── PR → 팀장 승인 → merge! │ ↓ main ────●───────────────────────────────────────────────────────────●────▶ (팀장: 직접 작업) │ ↑ └── feature/oliveyoung-sourcing (팀원 B) ─ 커밋 ─ PR → merge! 팀원: feature/xxx-plugin 또는 feature/xxx-sourcing 브랜치 생성 → PR → 팀장 승인 후 merge 팀장: main 브랜치에서 직접 작업 (PR 없음)

핵심 용어: 커밋(Commit) = 저장, 푸시(Push) = GitHub에 업로드, 풀(Pull) = GitHub에서 다운로드, 코드 검토 요청 = 팀장에게 "코드 확인해주세요" 보내기, 머지(Merge) = 브랜치를 합치기

3

로컬 DB + 환경 설정

ℹ️
우리 프로젝트의 데이터베이스
이 프로젝트는 PostgreSQL 16 (pgvector 확장 포함) 데이터베이스를 사용합니다.
Docker가 PostgreSQL을 자동으로 띄워주므로, 직접 설치할 필요 없습니다.
아래 순서대로 하면 됩니다.

3-1. Docker Desktop 실행 확인

Docker Desktop이 실행 중이어야 합니다. 작업 표시줄(우측 하단)에 고래 아이콘이 보여야 합니다.

터미널 — Docker 실행 확인
docker info # "Server: Docker Desktop" 이 보이면 OK
⚠️
"Cannot connect to the Docker daemon" 에러가 나오면?
→ Docker Desktop 앱을 실행하세요. 시작 메뉴에서 "Docker Desktop" 검색 → 클릭
→ 실행 후 30초~1분 기다린 후 다시 시도하세요. Docker는 시작에 시간이 걸립니다.
→ 그래도 안 되면 PC를 재시작하세요.

3-2. DB 컨테이너 실행

backend 폴더에서 아래 명령어를 실행합니다. PostgreSQL DB가 Docker 안에서 자동으로 시작됩니다.

터미널 (backend 폴더에서)
cd backend # 아직 backend 폴더가 아니라면 docker compose up -d # DB 컨테이너를 백그라운드로 실행
✅
성공하면 이런 메시지가 나옵니다:
Container ygs-test-db ... Started 또는 Container ygs-test-db ... Running

DB가 정상 실행 중인지 확인:

터미널 — DB 상태 확인
docker compose ps # STATE가 "running (healthy)" 이면 OK
⚠️
STATE가 "starting" 이면?
→ 10초 기다린 후 다시 docker compose ps를 실행하세요. DB 초기화에 시간이 걸릴 수 있습니다.

"port is already allocated" 에러가 나오면?
→ 포트 5433을 다른 프로그램이 사용 중입니다. 아래 명령으로 확인 후 해당 프로그램을 종료하세요:
netstat -ano | findstr :5433 또는 기존 컨테이너가 남아있을 수 있으므로: docker compose down && docker compose up -d

3-3. 환경변수 파일 복사

서버가 DB에 접속할 때 사용하는 설정 파일입니다. 예제 파일을 그대로 복사하면 됩니다.

터미널 (backend 폴더에서)
cp .env.example .env # 수정할 것 없음, 그대로 사용
ℹ️
.env 파일 내용 (참고용 — 수정하지 마세요)
WRITE_DB_HOST=localhost — DB가 내 컴퓨터에서 돌아감
WRITE_DB_PORT=5433 — Docker가 5433 포트를 사용
WRITE_DB_NAME=test_little_boy — 데이터베이스 이름
WRITE_DB_USER=test_user — DB 접속 아이디
WRITE_DB_PASSWORD=test_password — DB 접속 비밀번호
이 값들은 docker-compose.yaml과 맞춰져 있으므로 절대 수정하지 마세요.

3-4. DB 마이그레이션 (테이블 생성)

DB 컨테이너를 처음 띄우면 빈 데이터베이스입니다. Alembic 마이그레이션을 실행해서 필요한 테이블을 만들어야 합니다.

터미널 (backend 폴더에서, 가상환경 활성화 상태)
# 가상환경이 활성화 안 되어 있다면 먼저: source .venv/Scripts/activate # (.venv) 가 앞에 보여야 함 # Alembic 마이그레이션 실행 (테이블 생성) alembic upgrade head # 모든 테이블을 최신 상태로 생성
✅
성공하면 이런 메시지가 나옵니다:
INFO [alembic.runtime.migration] Running upgrade xxx -> yyy, ...
여러 줄이 나오고 에러 없이 끝나면 성공!
⚠️
"alembic: command not found" 에러?
→ 가상환경이 활성화되지 않은 상태입니다. source .venv/Scripts/activate 먼저 실행하세요.
→ 터미널 앞에 (.venv) 가 보이는지 확인하세요.

"Connection refused" 또는 "could not connect to server" 에러?
→ DB 컨테이너가 아직 안 뜬 상태입니다. docker compose ps로 확인 후 STATE가 "running (healthy)"인지 확인하세요.
→ "starting" 상태라면 10~20초 더 기다리세요.

"ModuleNotFoundError" 에러?
→ 2단계의 uv pip install -e .을 안 한 상태입니다. 2단계로 돌아가세요.

3-5. DB 연결 최종 확인

모든 설정이 잘 되었는지 최종 확인합니다.

터미널 (backend 폴더에서)
# DB에 직접 접속해서 테이블 목록 확인 docker exec -it ygs-test-db psql -U test_user -d test_little_boy -c "\dt" # 테이블 목록이 나오면 성공! 나가려면: # \q 입력 (위 명령은 자동으로 종료됩니다)
✅
테이블 목록이 보이면 DB 설정 완료!
samba_product, samba_order, samba_channel 등의 테이블이 보이면 정상입니다.

DB 트러블슈팅 — 자주 겪는 문제와 해결법

DB 관련 에러가 나면 여기서 찾아보세요. 대부분의 문제는 아래 방법으로 해결됩니다.

🔴 "Connection refused" — 서버 실행 시 DB 연결 실패

백엔드 서버를 실행했는데 DB 연결 에러가 나는 경우

원인: DB 컨테이너가 꺼져있음

PC를 재시작하거나 Docker Desktop을 껐다 켜면, DB 컨테이너도 같이 꺼집니다.

해결법
# 1. DB 컨테이너 상태 확인 cd backend docker compose ps # 2. 컨테이너가 없거나 꺼져있으면 다시 실행 docker compose up -d # 3. healthy 될 때까지 10초 기다린 후 서버 재실행
💡
매일 아침 개발 시작할 때 Docker Desktop을 켜고 docker compose up -d를 습관처럼 실행하세요!

🔴 "alembic" 관련 에러 모음

"alembic: command not found"

→ 가상환경이 활성화되지 않은 상태입니다.

해결법
source .venv/Scripts/activate # (.venv)가 앞에 나타나야 함 alembic upgrade head # 다시 실행

"Target database is not up to date" 또는 "Can't locate revision"

→ main에서 새 마이그레이션이 추가되었는데, 내 로컬 DB에는 아직 반영 안 된 경우입니다.

해결법
# 마이그레이션을 최신으로 업그레이드 alembic upgrade head # 그래도 안 되면 DB를 처음부터 다시 만들기 (아래 "DB 초기화" 참고)

"FAILED: Multiple head revisions"

→ 마이그레이션 충돌입니다. 팀장에게 알려주세요. 팀원이 해결할 필요 없습니다.

🟡 git pull 했더니 서버가 안 뜸

최신 코드를 받은 후 서버 실행 시 DB 관련 에러가 나는 경우. 다른 팀원이나 팀장이 DB 스키마를 변경했을 수 있습니다.

해결법 — pull 후 마이그레이션 실행
# 가상환경 활성화 확인 source .venv/Scripts/activate # 새로 추가된 패키지 설치 (혹시 모르니) uv pip install -e . # 마이그레이션 실행 (새 테이블/컬럼 반영) alembic upgrade head # 서버 다시 실행
💡
습관으로 만드세요: git pull 후에는 항상 alembic upgrade head를 실행!
Claude Code에게 "최신 코드 가져와줘"라고 하면 자동으로 해주기도 합니다.

🔄 최후의 수단: DB 완전 초기화

위 방법으로 해결이 안 되면, DB를 완전히 삭제하고 처음부터 다시 만드세요. 로컬 개발 DB이므로 삭제해도 괜찮습니다.

DB 완전 초기화 (backend 폴더에서)
# 1. 기존 DB 컨테이너 + 데이터 완전 삭제 docker compose down -v # -v 옵션이 데이터까지 삭제 # 2. 새 DB 컨테이너 생성 docker compose up -d # 3. DB가 healthy 될 때까지 10초 대기 후 마이그레이션 alembic upgrade head # 4. 서버 실행해서 확인

🛠️ 유용한 DB 관련 명령어 모음

DB 컨테이너 시작

docker compose up -d

DB 컨테이너 상태 확인

docker compose ps

DB 컨테이너 로그 확인 (에러 원인 파악)

docker compose logs test-db

DB 컨테이너 중지

docker compose down

마이그레이션 최신 적용

alembic upgrade head

현재 마이그레이션 상태 확인

alembic current

DB에 직접 접속해서 SQL 실행

docker exec -it ygs-test-db psql -U test_user -d test_little_boy

DB 완전 초기화 (데이터 포함 삭제)

docker compose down -v && docker compose up -d && alembic upgrade head

🗃️ DB 컬럼/테이블 추가 시 체크리스트
⚠️
모델(model.py) 필드를 추가/변경하면 반드시 마이그레이션 파일을 생성해야 합니다.
마이그레이션 없이 push하면 프로덕션 DB에 컬럼이 없어서 서버 오류가 발생합니다.
  • ✓model.py 필드 추가/변경
  • ✓cd backend && python scripts/check_migrations.py 실행 — 누락 검증
  • ✓alembic revision --autogenerate -m "변경 내용" 마이그레이션 파일 생성
  • ✓생성된 마이그레이션 파일 내용 검토 (자동생성 오류 확인)
  • ✓로컬에서 alembic upgrade head 적용 확인
  • ✓마이그레이션 파일을 커밋에 포함하여 push

머지 규칙 — 팀장 전용

팀원 PR을 머지하기 전에 반드시 공유 파일 변경 여부를 확인해야 합니다.
컨플릭트가 없어도 안전하지 않습니다 — Git은 한쪽만 변경하면 자동 채택하므로 팀장 코드가 통째로 유실될 수 있습니다.

🚨
컨플릭트 없음 ≠ 안전
팀원이 공유 파일을 수정했지만 팀장과 동일한 줄에 손대지 않으면 Git은 자동으로 팀원 내용을 채택합니다. 팀장이 공들여 만든 로직이 통째로 사라질 수 있습니다.
🔍 머지 전 필수 체크 절차
1

변경 파일 목록 확인

터미널
git diff main...팀원브랜치 --name-only

변경된 파일 목록이 나옵니다. 담당 플러그인 파일 외 다른 파일이 있으면 즉시 멈추세요.

2

허용 파일 vs 금지 파일

허용 (본인 담당만)
  • plugins/markets/본인마켓.py
  • plugins/sourcing/본인소싱처.py
  • proxy/본인마켓.py
금지 (머지 불가)
  • plugins/__init__.py
  • plugins/market_base.py
  • plugins/sourcing_base.py
  • collector/ 폴더 전체
  • 다른 팀원 담당 파일
  • 기타 공유 파일
3

공유 파일 변경 감지 시

diff 내용 확인
git diff main...팀원브랜치 -- 문제파일경로

diff를 확인하고 의도치 않은 변경이라면 머지하지 말고 팀원에게 해당 파일 변경을 되돌려달라고 요청하세요.

코드 작성 규칙

코드를 작성할 때 반드시 지켜야 할 규칙들입니다. Claude Code가 이 규칙에 맞게 도와줍니다.

절대 금지
  • ❌ any 타입 사용 (TypeScript)
  • ❌ 스피너(loading spinner) 사용
  • ❌ 팝업창 — 반드시 모달창으로 구현
  • ❌ 텍스트 2행 초과 — UI 레이아웃 조정 필요
  • ❌ 삭제(마켓삭제)와 판매중지(SUSPENSION/STOP) 혼동
필수 준수
  • ✅ 언어: TypeScript, Python (타입 명시)
  • ✅ 코드 주석: 한국어
  • ✅ 커밋 메시지: 한국어
  • ✅ 변수명/함수명: 영어 (camelCase)
  • ✅ 들여쓰기: 2칸 스페이스
🔢 숫자 표시 규칙 — UI에 표시되는 모든 숫자
⚠️
UI에 표시되는 모든 숫자는 예외 없이 천 단위 콤마를 적용해야 합니다.
1000 → 1,000 / 50000 → 50,000 / 1234567 → 1,234,567
대상사용 방법예시
UI 렌더링 (가격, 건수, 수량, 재고 등) fmtNum(value) {fmtNum(count)}
로그 메시지 (addLog) .toLocaleString() `${count.toLocaleString()}건`
제외 대상 백분율(%), 소수점 시간, 페이지번호, 배열 인덱스
올바른 사용 예시
// import 추가 import { fmtNum } from '@/lib/samba/styles' // 올바른 사용 <span>{fmtNum(count)}건</span> <span>{fmtNum(price)}원</span> // 잘못된 사용 (금지) <span>{count}건</span> // ❌ <span>{total}원</span> // ❌
🗄️ DB 세션 규칙
ℹ️
DB 세션은 반드시 Read/Write 분리 사용
• 데이터 수정/생성/삭제 → get_write_session_dependency()
• 조회 전용 → get_read_session_dependency()
두 세션을 절대 혼용하지 말 것. 혼용 시 데이터 정합성 문제 발생.

👑 팀장이 할 일
# 할 일 설명 시점
1 코드 검토 & 반영 팀원이 올린 코드를 검토하고, 문제 없으면 반영 요청 올라올 때
2 스킬 노하우 축적 git pull 시 post-merge hook이 자동 실행 → 머지된 코드를 분석해서 스킬 파일에 노하우 반영 자동 (머지 후)
3 할당표 관리 담당자 배정, 상태 변경 (할당 대기 → 할당 → 완료) 수시
4 공유 파일 수정 __init__.py, base.py, 확장앱 등 팀원이 못 건드리는 파일은 팀장이 직접 수정 필요 시
5 충돌 해결 팀원 간 충돌 발생 시 화면 공유로 같이 해결 거의 없음
6 배포 main 브랜치 코드를 실서버에 반영 (팀원은 배포 권한 없음) 기능 완성 후
📌
팀장의 일상 루틴
GitHub에서 검토 요청 확인 → 코드 확인 → 반영 → git pull (스킬 자동 업데이트) → 할당표 상태 변경
