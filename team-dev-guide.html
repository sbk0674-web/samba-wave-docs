이것부터 이해하기

코드를 쓰기 전에, 우리 프로젝트가 어떤 구조인지 먼저 이해합시다. 여기서 비유를 통해 설명하니 끝까지 읽어주세요.

🎯 우리가 만드는 것
💡 비유로 이해하기

우리 시스템은 "번역기"와 비슷합니다.

소싱처(무신사, 올리브영 등)에서 상품 정보를 가져와서,
판매처(쿠팡, 스마트스토어 등)의 형식에 맞게 변환해서 올려주는 겁니다.

예를 들어, 무신사에서 "나이키 에어맥스"를 가져와서 → 쿠팡 형식으로 바꿔서 → 쿠팡에 등록.
이 과정을 자동으로 해주는 게 우리 시스템입니다.

🧩 플러그인이 뭔가요?
💡 비유로 이해하기

플러그인은 "USB 장치"와 같습니다.

컴퓨터 본체(우리 시스템)는 이미 완성되어 있고,
마우스(쿠팡), 키보드(스마트스토어), 웹캠(11번가)처럼
USB만 꽂으면 바로 인식되죠? 드라이버 설치 같은 거 필요 없이요.

우리 플러그인도 마찬가지입니다. 파일 하나 만들면 자동으로 인식됩니다.
다른 파일을 수정할 필요가 전혀 없어요.

🏪 판매처(마켓) 플러그인

쿠팡, 스마트스토어, 11번가 같은 곳에 상품을 등록하는 코드

역할: 상품 데이터를 해당 마켓 형식으로 변환하고 전송

📦 소싱처 플러그인

무신사, 올리브영 같은 곳에서 상품을 가져오는 코드

역할: 상품 검색, 상세 조회, 가격/재고 갱신

👥 왜 10명이 동시에 해도 괜찮은가?
💡 비유로 이해하기

아파트를 생각해보세요. 10명이 각자 다른 호수에서 인테리어 공사를 합니다.

101호에서 벽지를 바꾸든, 102호에서 바닥을 교체하든 서로 영향이 없죠?
하지만 복도(공용부분)를 동시에 공사하면 부딪히겠죠.

우리 프로젝트도 같습니다:
• 각자 만드는 플러그인 파일 = 각자의 호수 (충돌 없음)
• 공유 파일(__init__.py, base.py) = 복도 (건드리면 안 됨)

용어 사전

이 문서에서 계속 나오는 단어들입니다. 모르면 여기로 돌아오세요.

용어 초등학생한테 설명한다면 예시
Git "되돌리기"가 무한대인 저장 시스템. 코드의 모든 변경 이력을 기록해줌 워드 문서의 "변경 내용 추적"
Repository(저장소) 프로젝트 코드가 들어있는 폴더. 줄여서 "레포" samba-wave 폴더 전체
Branch(브랜치) 원본을 복사해서 따로 작업하는 공간. "평행 세계" 같은 것 feature/tmon-plugin
main 브랜치 "진짜 원본". 여기를 직접 수정하면 안 됨 출판된 책의 원고
Commit(커밋) "저장" 버튼. 지금까지의 변경을 기록으로 남김 게임에서 "세이브" 하는 것
Push(푸시) 내 컴퓨터의 저장 내용을 GitHub(인터넷)에 올리기 구글 드라이브에 업로드
Pull(풀) GitHub(인터넷)의 최신 내용을 내 컴퓨터로 가져오기 구글 드라이브에서 다운로드
Merge(머지) 두 개의 브랜치를 합치기 복사본의 변경을 원본에 반영
PR (Pull Request) "제 코드 좀 확인해주세요" 하고 팀장에게 보내는 요청 과제물 제출 후 선생님 검사
충돌(Conflict) 두 사람이 같은 줄을 동시에 수정해서 Git이 어떤 걸 써야 할지 모르는 상태 같은 문장을 두 사람이 다르게 고침
Plugin(플러그인) 본체를 수정하지 않고 기능을 추가하는 부품 USB 장치
Proxy(프록시) 외부 사이트의 API를 대신 호출해주는 중간 역할 대리인, 통역사
API 프로그램끼리 대화하는 규칙. "이렇게 보내면 이렇게 답해줄게" 자판기의 버튼과 음료 나오는 구멍

개발 환경 세팅

한 번만 하면 되는 초기 세팅입니다. 순서대로 하나씩 따라하세요.

ℹ️
사전 설치 필요 아래 프로그램이 설치되어 있어야 합니다. 없으면 팀장에게 알려주세요.
• Git — git-scm.com 에서 다운로드. 설치 시 기본값 그대로 "Next" 연타하면 됨
• Python 3.12 — python.org 에서 다운로드. 설치 시 "Add Python to PATH" 반드시 체크!
• uv (파이썬 패키지 관리자) — pip보다 10배 빠른 패키지 설치 도구. Python 설치 후 터미널에서 아래 명령어 실행:
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex" 설치 후 터미널을 껐다 다시 열고, uv --version 입력해서 버전이 나오면 성공
• Docker Desktop — docker.com 에서 다운로드. 로컬 DB를 띄우는 데 필요
• Claude Code (AI 코딩 도우미) — 코드 작성, Git 관리, 파일 생성을 대화로 처리. 팀장이 제공하는 설치 가이드로 설치
1

GitHub에서 코드 가져오기 (클론)

프로젝트 코드를 내 컴퓨터에 복사합니다. 딱 한 번만 하면 됩니다.

터미널 (Git Bash)
# 1. 원하는 폴더로 이동 (예: workspace 폴더) cd ~/workspace # 2. GitHub에서 코드 다운로드 git clone https://github.com/sbk0674-web/samba-wave.git # 3. 프로젝트 폴더로 이동 cd samba-wave
💡
Git Bash 여는 법 Windows에서 바탕화면 우클릭 → "Git Bash Here" 클릭.
또는 Claude Code에서 바로 작업할 수도 있습니다. 프로젝트 클론해줘 라고 말해도 됩니다.
2

백엔드 환경 만들기

파이썬 가상환경을 만들고, 필요한 패키지를 설치합니다.

터미널 (Git Bash)
# 1. backend 폴더로 이동 cd backend # 2. 파이썬 가상환경 만들기 (가상환경 = 이 프로젝트 전용 파이썬) uv venv # 3. 가상환경 켜기 (Windows) source .venv/Scripts/activate # 4. 필요한 패키지 설치 uv pip install -e . uv pip install -e .[dev]
⚠️
가상환경이 켜져있는지 확인하는 법 터미널 맨 앞에 (.venv)가 보이면 켜져있는 겁니다.
안 보이면 source .venv/Scripts/activate를 다시 입력하세요.
3

로컬 데이터베이스 띄우기 + 환경 설정

명령어 2줄이면 끝. DB 실행 + 환경 설정이 한번에 됩니다.

터미널 (backend 폴더에서)
# 1. 로컬 DB 띄우기 (Docker Desktop이 실행 중이어야 함) docker compose up -d # 2. 환경 설정 파일 복사 (수정할 거 없음, 그대로 사용) cp .env.example .env
✅
이게 전부입니다! DB 아이디/비밀번호/주소가 .env.example에 이미 세팅되어 있어서 수정할 필요 없습니다.
⚠️
Docker Desktop이 없다면? docker.com에서 다운로드 → 설치 → 실행 후 다시 시도하세요.
4

백엔드 실행해서 확인

모든 세팅이 끝났으면 서버를 실행해봅니다.

터미널
# backend 폴더에서 실행 cd backend .venv/Scripts/python.exe run.py
✅
성공 확인 로그에 [플러그인] 마켓 18개, 소싱 3개 등록 완료가 보이면 성공!
에러가 나면 팀장에게 스크린샷 보내주세요.
4

Git이 뭔가요?

💡 비유로 이해하기 — "Google Docs"

Google Docs를 생각해보세요. 여러 사람이 동시에 문서를 편집할 수 있고,
"버전 기록"에서 누가 뭘 바꿨는지 볼 수 있죠?

Git도 똑같습니다. 다만 코드 파일 전용이고, 더 정밀합니다.

Google Docs와 다른 점 하나:
Google Docs는 실시간으로 합쳐지지만,
Git은 "내가 원할 때" 합칩니다. 이게 브랜치라는 개념입니다.

🌿 브랜치 = 평행 세계

브랜치를 만드는 건 "평행 세계를 하나 만든다"고 생각하세요.

브랜치 개념도
┌── feature/tmon (A가 작업 중) ──── 커밋 ── 커밋 ── merge! │ ↓ main ────●────────────────────────────────────────────────────●────▶ │ ↑ └── feature/oliveyoung (B가 작업 중) ─ 커밋 ─ merge! ● = 시작 지점 ▶ = 현재 진행 중 A와 B는 각자 다른 "세계"에서 작업하다가, 완료되면 main에 합침
  • 1 main 브랜치는 "진짜 세계"입니다. 여기 있는 코드가 실제로 돌아가는 코드예요.
  • 2 feature 브랜치는 "연습 세계"입니다. 여기서 마음껏 실험하고, 완성되면 진짜 세계에 합칩니다.
  • 3 각자 다른 브랜치에서 작업하기 때문에, 다른 사람 작업에 영향을 주지 않습니다.
5

매일 하는 Git 루틴

매일 아침 작업을 시작하기 전에 반드시 하는 과정입니다. 이걸 안 하면 옛날 코드 위에서 작업하게 됩니다.

🌅 아침 — 최신 코드 가져오기
💬 Claude Code에 말하기
"main에서 최신 코드 가져와서 내 브랜치에 merge해줘"

한 줄이면 Claude Code가 알아서 checkout, pull, merge를 다 해줍니다.

직접 터미널로 하고 싶다면
1

main 브랜치로 이동

먼저 "진짜 원본"이 있는 곳으로 갑니다.

터미널
git checkout main
⚠️
에러가 나면? → 아직 저장 안 한 변경이 있다는 뜻입니다.
먼저 git stash를 입력하세요 (임시 보관). 나중에 git stash pop으로 복구.
2

GitHub에서 최신 코드 다운로드

다른 사람이 밤사이에 합친(merge한) 코드를 가져옵니다.

터미널
git pull origin main
3

내 브랜치로 돌아가서 최신 코드 합치기

내 작업 브랜치에도 최신 코드를 반영합니다.

터미널
# 내 브랜치로 돌아가기 git checkout feature/tmon-plugin # main의 최신 코드를 내 브랜치에 합치기 git merge main
✅
이제 내 브랜치에도 다른 사람들의 최신 코드가 반영되었습니다.
안심하고 작업을 시작하세요!
🆕 처음 시작할 때 — 브랜치 만들기

프로젝트에 처음 참여할 때, 또는 새로운 작업을 시작할 때 한 번 합니다.

💬 Claude Code에 말하기
"쿠팡 플러그인 작업할 브랜치 만들어줘"

Claude Code가 main에서 최신 코드를 가져온 후, 브랜치 이름 규칙에 맞게 자동으로 만들어줍니다.

직접 터미널로 하고 싶다면
터미널
# 1. 먼저 main에서 최신 코드 가져오기 git checkout main git pull origin main # 2. 새 브랜치 만들기 (main을 기반으로) git checkout -b feature/내마켓이름-plugin # 예시들: git checkout -b feature/coupang-plugin # 판매처: 쿠팡 git checkout -b feature/oliveyoung-sourcing # 소싱처: 올리브영 git checkout -b feature/interpark-plugin # 판매처: 인터파크
작업 종류 브랜치 이름 규칙 예시
판매처 feature/마켓이름-plugin feature/tmon-plugin
소싱처 feature/소싱처이름-sourcing feature/oliveyoung-sourcing
버그수정 fix/설명 fix/coupang-auth-error
6

커밋하기 (저장하기)

💡 비유로 이해하기 — "게임 세이브"

RPG 게임에서 보스전 전에 세이브하잖아요?
커밋도 마찬가지입니다. 작업하다가 "여기까지는 잘 됐다!" 하면 커밋(세이브)하세요.
실수해도 이전 세이브 지점으로 돌아갈 수 있습니다.

📝 Claude Code로 커밋하기
💬 Claude Code에 말하기
/commit

또는

💬 Claude Code에 말하기
"지금까지 작업한 거 커밋해줘"

Claude Code가 변경 파일 확인 → 내 파일만 선택 → 커밋 메시지 작성까지 자동으로 해줍니다.

🚨
왜 git add .이 위험한가요? git add .은 폴더 안의 모든 파일을 추가합니다.
내가 모르는 사이에 .env(비밀번호), 다른 사람 파일, 불필요한 파일이 함께 올라갈 수 있습니다.
반드시 내 파일만 이름을 지정해서 추가하세요.

Claude Code를 쓰면 이런 실수를 할 일이 없습니다. Claude Code가 변경된 파일을 보여주고, 내 파일만 골라서 커밋해줍니다.
직접 터미널로 하고 싶다면
1

변경 사항 확인하기

"내가 뭘 바꿨지?" 확인합니다.

터미널
git status

그러면 이런 결과가 나옵니다:

결과 예시
On branch feature/tmon-plugin Untracked files: ← "새로 만든 파일"이라는 뜻 backend/backend/domain/samba/plugins/markets/tmon.py backend/backend/domain/samba/proxy/tmon.py
2

파일을 스테이징하기 (커밋할 파일 선택)

"이 파일을 커밋에 포함시킬게요"라고 Git에게 알려줍니다.

✅ 올바른 방법 — 파일을 하나씩 지정
git add backend/backend/domain/samba/plugins/markets/tmon.py git add backend/backend/domain/samba/proxy/tmon.py
❌ 틀린 방법 — 전부 다 추가
# 절대 이렇게 하지 마세요!!! git add . git add -A
3

커밋하기 (저장!)

메시지와 함께 저장합니다. 메시지는 "뭘 했는지" 적어주세요.

터미널
git commit -m "[마켓] 쿠팡 플러그인 — transform 구현"

푸시 & PR (코드 올리기)

☁️ 푸시 — 내 코드를 GitHub에 올리기
💡 비유로 이해하기

커밋 = 내 컴퓨터에 저장 (USB에 저장)
푸시 = 인터넷에 업로드 (구글 드라이브에 올리기)

커밋만 하면 내 컴퓨터에만 있습니다. 푸시해야 다른 사람이 볼 수 있어요.

💬 Claude Code에 말하기
"GitHub에 푸시해줘"

Claude Code가 현재 브랜치를 확인하고, 올바른 remote에 push해줍니다. 처음 푸시할 때 upstream 설정도 자동으로 해줍니다.

직접 터미널로 하고 싶다면
터미널
# 내 브랜치를 GitHub에 올리기 git push origin feature/tmon-plugin # 처음 푸시할 때 이런 메시지가 나올 수 있어요: # "To push the current branch and set the remote as upstream..." # 그러면 이렇게 입력: git push --set-upstream origin feature/tmon-plugin
📬 PR — "제 코드 확인해주세요" 요청
💡 비유로 이해하기

학교 과제를 다 했으면 선생님한테 검사 맡잖아요?
PR = "선생님, 과제 다 했습니다. 확인해주세요!" 하는 거예요.
팀장이 코드를 보고, 괜찮으면 main에 합쳐(merge)줍니다.

💬 Claude Code에 말하기
"PR 만들어줘"

Claude Code가 push + PR 생성까지 한번에 해줍니다. 제목과 본문도 변경 내용을 분석해서 자동으로 작성해줍니다.

직접 GitHub 웹사이트에서 하고 싶다면
1

GitHub 웹사이트 열기

브라우저에서 우리 프로젝트 GitHub 페이지로 갑니다.

2

"Compare & pull request" 버튼 클릭

푸시하면 GitHub에 노란색 배너가 뜹니다. 그 버튼을 누르세요.

3

PR 내용 작성

PR 작성 예시
제목: [마켓] 쿠팡 플러그인 추가 본문: ## 작업 내용 - 쿠팡 마켓 플러그인 구현 (plugins/markets/coupang.py) - 쿠팡 HTTP 클라이언트 구현 (proxy/coupang.py) - 상품 등록/수정 API 연동 완료 ## 테스트 - [x] 서버 재시작 시 플러그인 자동 등록 확인 - [x] transform 함수 데이터 변환 확인 - [ ] 실제 API 호출 테스트 (테스트 계정 필요)

"Create Pull Request" 클릭 → 팀장 리뷰 대기

팀장이 코드를 검토하고, 수정 요청이나 승인을 해줍니다.

충돌 해결하기

🎉
좋은 소식! 각자 다른 파일을 만들기 때문에 충돌이 거의 발생하지 않습니다.
하지만 만약을 위해 알아두세요.
⚡ 충돌이 생기면
💬 Claude Code에 말하기
"충돌 해결해줘"

Claude Code가 충돌 파일을 찾아서 내용을 분석하고, 어떤 코드를 남길지 판단해서 해결해줍니다.

직접 터미널로 하고 싶다면
⚡ 충돌이 생기면 이렇게 보입니다
충돌이 발생한 파일 (예시)
<<<<<<< HEAD 내가 수정한 코드 market_type = "tmon" ======= 다른 사람이 수정한 코드 market_type = "tmon_v2" >>>>>>> main
  • 1 <<<<<<< HEAD ~ ======= 사이 = 내 코드
  • 2 ======= ~ >>>>>>> main 사이 = 상대방 코드
  • 3 둘 중 맞는 코드를 남기고, 나머지와 <<< === >>> 표시를 전부 삭제
🔧 해결 순서
터미널
# 1. 충돌 파일을 에디터에서 열기 (에디터에서 직접 수정) # → 충돌 부분을 색깔로 보여주고 # → "Accept Current Change" / "Accept Incoming Change" 버튼이 나옴 # → 맞는 걸 클릭하면 됨 # 2. 수정 완료 후 저장 # 3. 수정한 파일을 스테이징 git add 충돌파일이름.py # 4. 충돌 해결 커밋 git commit -m "충돌 해결 — main merge"
⚠️
잘 모르겠으면? 절대 대충 해결하지 마세요. 팀장에게 화면 공유하면서 같이 해결하세요.

  • 3MarketPlugin 또는 SourcingPlugin을 상속했는지 확인
  • 4로그에 로드 실패 메시지가 있는지 확인 (import 에러 등)

절대 금지 사항

🚨
아래 행위는 팀 전체에 피해를 줍니다. 반드시 지켜주세요.
# 금지 행위 왜 위험한가 대신 이렇게
1 git add . 사용 .env(비밀번호), 다른 사람 파일, 임시 파일이 전부 올라감 git add 내파일.py
2 main에 직접 커밋 다른 9명에게 즉시 영향. 버그가 있으면 전원 작업 불가 feature 브랜치에서 작업 후 PR
3 git push --force 다른 사람이 올린 코드가 통째로 사라짐. 복구 불가 일반 git push만 사용
4 공유 파일 수정 __init__.py, market_base.py, sourcing_base.py를 수정하면 전원에게 영향 팀장에게 먼저 말하기
5 다른 사람 파일 수정 충돌 발생 + 상대방 작업 덮어쓸 위험 해당 담당자에게 요청
6 이름 중복 market_type/site_name이 겹치면 하나가 덮어쓰기 됨 아래 "사용 중인 이름" 확인
7 .env 커밋 비밀번호가 GitHub에 올라가면 전세계에 공개. 해킹 위험 .gitignore가 막아주지만, 직접 추가하지 말 것

자주 묻는 질문 (FAQ)

Q. 내가 어떤 브랜치에 있는지 어떻게 알아요?

Claude Code에서는 항상 현재 브랜치가 표시되어 있습니다. 별도로 확인할 필요 없어요.

직접 터미널로 확인하고 싶다면
git branch # 별표(*)가 붙은 게 지금 있는 브랜치 # main # * feature/tmon-plugin ← 여기 있음
Q. 실수로 main에 커밋해버렸어요!
💬 Claude Code에 말하기
"방금 커밋 되돌려줘"

Claude Code가 커밋을 되돌리고, 올바른 브랜치에서 다시 커밋하도록 안내해줍니다.

직접 터미널로 하고 싶다면

아직 push 안 했다면:

# 커밋을 되돌리기 (코드는 유지, 커밋만 취소) git reset --soft HEAD~1 # 새 브랜치 만들고 거기서 다시 커밋 git checkout -b feature/내마켓-plugin git add 내파일.py git commit -m "메시지"

이미 push 했다면 → 팀장에게 즉시 연락하세요.

Q. git pull 했더니 이상한 에디터가 열렸어요!

Claude Code를 쓰면 이런 상황이 발생하지 않습니다. Claude Code가 Git 명령어를 대신 실행하기 때문에 vim 에디터 같은 문제를 걱정할 필요가 없습니다.

직접 터미널에서 이 상황이 발생했다면

merge 커밋 메시지를 작성하라는 뜻입니다.

# vim 에디터가 열렸을 때: # 1. :q 입력하고 Enter (그냥 나가기) # 또는 # 1. Esc 키 누르기 # 2. :wq 입력하고 Enter (저장하고 나가기)
Q. 서버가 안 켜져요! 에러 메시지가 나와요

자주 발생하는 원인들:

  • 1가상환경이 꺼져있음 → source .venv/Scripts/activate 실행
  • 2.env 파일이 없음 → 팀장에게 파일 요청
  • 3import 에러 → 내 플러그인 코드에 오타가 있을 수 있음. 에러 메시지 확인
  • 4DB 연결 실패 → VPN이 꺼져있거나 .env의 DB 주소가 틀린 경우
Q. transform 함수에 어떤 필드를 넣어야 하나요?

해당 마켓의 API 개발 문서를 봐야 합니다.
마켓마다 요구하는 필드가 다릅니다 (상품명, 가격, 카테고리, 배송비, 이미지 등).

기존 구현체를 참고하세요:
• 간단한 예: plugins/markets/elevenst.py (71줄)
• 복잡한 예: plugins/markets/coupang.py (143줄)

Q. 모르겠으면 어떡하나요?

1. Claude Code에 질문하기 — 코드 관련 질문은 대부분 해결됩니다
2. 이 가이드를 다시 읽어보기
3. 기존 코드 파일 열어서 참고하기 (복사 → 수정하면 됨!)
4. 팀 채팅방에 질문하기
5. 팀장에게 화면 공유하며 도움 요청
