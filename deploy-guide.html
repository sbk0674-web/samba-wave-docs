사전 준비

필요한 계정 및 도구

  • GCP(Google Cloud Platform) 계정 — 결제 수단 등록 필수
  • GitHub 계정 — 리포지토리 push 권한 보유
  • Vercel 계정 — vercel.com에서 GitHub으로 로그인
  • gcloud CLI 설치 — https://cloud.google.com/sdk/docs/install
  • Docker Desktop 설치 (로컬 빌드 테스트용)
  • uv 설치 — Python 패키지 관리자 pip install uv

GCP 초기 설정 (최초 1회)

# GCP 로그인 gcloud auth login # 프로젝트 설정 (프로젝트 ID 확인 후 입력) gcloud config set project YOUR_PROJECT_ID # 필요한 API 활성화 gcloud services enable run.googleapis.com gcloud services enable sqladmin.googleapis.com gcloud services enable artifactregistry.googleapis.com gcloud services enable secretmanager.googleapis.com # Artifact Registry 저장소 생성 (Docker 이미지 저장소) gcloud artifacts repositories create samba-wave \ --repository-format=docker \ --location=asia-northeast3 \ --description="SambaWave Docker images"
2

Cloud SQL 설정 (Read/Write 2개 인스턴스)

쓰기 인스턴스 생성

# 쓰기 전용 인스턴스 생성 gcloud sql instances create samba-wave-write \ --database-version=POSTGRES_15 \ --tier=db-f1-micro \ --region=asia-northeast3 \ --storage-size=10GB \ --storage-type=SSD \ --backup \ --backup-start-time=03:00 # 데이터베이스 생성 gcloud sql databases create sambawave \ --instance=samba-wave-write # 사용자 생성 gcloud sql users create samba_write_user \ --instance=samba-wave-write \ --password=YOUR_STRONG_PASSWORD

읽기 전용 인스턴스 생성 (Read Replica)

# 쓰기 인스턴스의 Read Replica 생성 gcloud sql instances create samba-wave-read \ --master-instance-name=samba-wave-write \ --region=asia-northeast3 \ --tier=db-f1-micro # Read Replica용 사용자 생성 (쓰기 인스턴스에서 복제됨) gcloud sql users create samba_read_user \ --instance=samba-wave-read \ --password=YOUR_READ_PASSWORD

Cloud SQL 연결 정보 확인

# 인스턴스 연결 이름 확인 (Cloud Run 연결 시 필요) gcloud sql instances describe samba-wave-write \ --format="value(connectionName)" # 출력 예: my-project:asia-northeast3:samba-wave-write # IP 주소 확인 gcloud sql instances describe samba-wave-write \ --format="value(ipAddresses[0].ipAddress)"
주의: Cloud SQL IP를 직접 사용하지 말고 Cloud SQL Auth Proxy 또는 Cloud Run의 Cloud SQL 직접 연결을 사용하세요. 연결 이름(connectionName)을 GitHub Secrets에 저장합니다.
3

GitHub Secrets 등록

GitHub 리포지토리 → Settings → Secrets and variables → Actions에서 아래 시크릿을 등록합니다.

Secret 이름값설명
GCP_SA_KEY 필수 서비스 계정 JSON 전체 Cloud Run/Artifact Registry 배포 권한
DB_WRITE_HOST 필수 쓰기 인스턴스 IP 또는 연결명 Cloud SQL 쓰기 인스턴스 호스트
DB_WRITE_USER 필수 samba_write_user 쓰기 DB 사용자명
DB_WRITE_PASSWORD 필수 (설정한 비밀번호) 쓰기 DB 비밀번호
DB_READ_HOST 필수 읽기 인스턴스 IP 또는 연결명 Cloud SQL 읽기 인스턴스 호스트
DB_READ_USER 필수 samba_read_user 읽기 DB 사용자명
DB_READ_PASSWORD 필수 (설정한 비밀번호) 읽기 DB 비밀번호
JWT_SECRET_KEY 필수 긴 랜덤 문자열 (32자 이상) JWT 토큰 서명 키

GCP 서비스 계정 생성 및 키 발급

# 서비스 계정 생성 gcloud iam service-accounts create samba-wave-deployer \ --display-name="SambaWave GitHub Actions Deployer" # 필요한 IAM 권한 부여 gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \ --member="serviceAccount:samba-wave-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com" \ --role="roles/run.admin" gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \ --member="serviceAccount:samba-wave-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com" \ --role="roles/artifactregistry.writer" gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \ --member="serviceAccount:samba-wave-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com" \ --role="roles/iam.serviceAccountUser" # JSON 키 파일 생성 → 이 내용을 GCP_SA_KEY 시크릿에 등록 gcloud iam service-accounts keys create gcp-sa-key.json \ --iam-account=samba-wave-deployer@YOUR_PROJECT_ID.iam.gserviceaccount.com
보안 주의: gcp-sa-key.json 파일은 GitHub Secrets에 등록 후 즉시 삭제하세요. 절대 Git 커밋 금지입니다.
4

GitHub Actions 워크플로우 (deploy-cloudrun.yml)

파일 위치: .github/workflows/deploy-cloudrun.yml

워크플로우 동작 순서

git push main → Actions 트리거 → GCP 인증 → Docker Build → Artifact Registry Push → Cloud Run 배포

워크플로우 핵심 설정 항목

# .github/workflows/deploy-cloudrun.yml 주요 설정 on: push: branches: [main] paths: - 'backend/**' # 백엔드 변경 시에만 트리거 env: PROJECT_ID: YOUR_GCP_PROJECT_ID REGION: asia-northeast3 SERVICE_NAME: samba-wave-backend REGISTRY: asia-northeast3-docker.pkg.dev # Cloud Run 배포 시 환경변수 주입 gcloud run deploy $SERVICE_NAME \ --set-env-vars DB_WRITE_HOST=${{ secrets.DB_WRITE_HOST }} \ --set-env-vars DB_WRITE_USER=${{ secrets.DB_WRITE_USER }} \ --set-env-vars DB_WRITE_PASSWORD=${{ secrets.DB_WRITE_PASSWORD }} \ --set-env-vars DB_READ_HOST=${{ secrets.DB_READ_HOST }} \ --set-env-vars DB_READ_USER=${{ secrets.DB_READ_USER }} \ --set-env-vars DB_READ_PASSWORD=${{ secrets.DB_READ_PASSWORD }} \ --set-env-vars JWT_SECRET_KEY=${{ secrets.JWT_SECRET_KEY }} \ --memory=2Gi \ --min-instances=1 \ --allow-unauthenticated
프론트엔드는 Actions 불필요: Vercel은 GitHub push를 감지해 자동 배포합니다. 백엔드 전용 변경 시 Actions만 실행되도록 paths 필터를 설정해 Actions 비용을 절약합니다.
5

DB 마이그레이션 (배포 전 필수)

최우선 규칙: CI/CD는 DB를 건드리지 않습니다. 마이그레이션은 반드시 수동으로 실행해야 합니다. 백엔드 배포 전에 반드시 먼저 실행하세요.

마이그레이션 파일 생성 (모델 변경 시)

# 백엔드 디렉토리에서 실행 cd backend # 마이그레이션 누락 여부 검증 (모델 변경 후 반드시 실행) .venv/Scripts/python.exe scripts/check_migrations.py # 마이그레이션 파일 자동 생성 alembic revision --autogenerate -m "컬럼 추가 설명" # 생성된 파일 검토 (alembic/versions/ 폴더) # upgrade/downgrade 함수가 올바른지 확인

프로덕션 DB에 마이그레이션 적용

# 방법 1: Cloud SQL Auth Proxy를 통한 로컬 연결 # Cloud SQL Auth Proxy 다운로드 후 실행 ./cloud-sql-proxy YOUR_PROJECT_ID:asia-northeast3:samba-wave-write & # 로컬 5432 포트로 연결되므로 localhost를 HOST로 사용 set WRITE_DB_HOST=127.0.0.1 set WRITE_DB_USER=samba_write_user set WRITE_DB_PASSWORD=YOUR_PASSWORD set WRITE_DB_NAME=sambawave alembic upgrade head # 방법 2: Cloud Shell에서 직접 실행 # GCP Console → Cloud Shell → 리포 클론 후 실행
성공 확인: "Running upgrade ... -> head" 메시지가 출력되면 성공입니다. 마이그레이션 이력은 DB의 alembic_version 테이블에서 확인할 수 있습니다.
6

Vercel 프론트엔드 배포 (최초 1회 설정)

Vercel 프로젝트 최초 연결

  • vercel.com 접속 → GitHub으로 로그인
  • Add New Project → samba-wave 리포지토리 선택
  • Root Directory: frontend 로 설정
  • Framework Preset: Next.js (자동 감지됨)
  • Environment Variables 섹션에서 아래 변수 추가
  • Deploy 클릭 → 2~3분 후 배포 완료

Vercel 환경변수 설정

# Vercel 대시보드 → 프로젝트 → Settings → Environment Variables NEXT_PUBLIC_API_URL = https://samba-wave-backend-XXXX-an.a.run.app # Cloud Run 배포 후 발급되는 서비스 URL
이후에는 git push main만 하면 Vercel이 자동으로 감지해서 재배포합니다. 별도 작업 불필요.
7

배포 후 확인 방법

백엔드 Health Check

# Cloud Run 서비스 URL 확인 gcloud run services describe samba-wave-backend \ --region=asia-northeast3 \ --format="value(status.url)" # 헬스체크 엔드포인트 (브라우저 또는 curl) https://samba-wave-backend-XXXX-an.a.run.app/docs https://samba-wave-backend-XXXX-an.a.run.app/health # Cloud Run 최신 배포 로그 확인 gcloud run services logs read samba-wave-backend \ --region=asia-northeast3 \ --limit=50

GitHub Actions 배포 상태 확인

  • GitHub 리포지토리 → Actions 탭 → 최신 워크플로우 실행 확인
  • 각 Step (Build, Push, Deploy) 녹색 체크 확인
  • 실패 시 해당 Step 클릭 → 상세 로그 확인
  • Cloud Run Console → 서비스 → 최신 리비전 상태 확인 (녹색 = 정상)

프론트엔드 확인

  • Vercel 대시보드 → 프로젝트 → 최신 배포 Status 확인 (Ready = 성공)
  • 배포 URL 접속 → https://your-app.vercel.app/samba
  • 로그인 후 기본 페이지 로딩 확인
  • 브라우저 개발자도구 → Network → API 호출 응답 200 확인
8

롤백 방법

Cloud Run 롤백 (백엔드)

# 현재 서비스의 리비전 목록 확인 gcloud run revisions list \ --service=samba-wave-backend \ --region=asia-northeast3 # 이전 리비전으로 트래픽 100% 전환 gcloud run services update-traffic samba-wave-backend \ --region=asia-northeast3 \ --to-revisions=samba-wave-backend-PREVIOUS_REVISION=100 # 확인 gcloud run services describe samba-wave-backend \ --region=asia-northeast3 \ --format="value(spec.traffic)"

Vercel 롤백 (프론트엔드)

  • Vercel 대시보드 → 프로젝트 → Deployments 탭
  • 롤백할 이전 배포 항목 찾기
  • 해당 배포의 점 세 개 메뉴(⋯) → Promote to Production 클릭
  • 즉시 적용됨 (빌드 재실행 없음)

DB 마이그레이션 롤백

# 한 단계 이전으로 되돌리기 alembic downgrade -1 # 특정 리비전으로 되돌리기 alembic downgrade REVISION_ID # 현재 리비전 확인 alembic current # 리비전 히스토리 확인 alembic history
주의: DB 롤백은 데이터 손실 가능성이 있습니다. downgrade 함수가 올바르게 작성되어 있는지 먼저 확인하고, 가능하면 DB 스냅샷 백업 후 진행하세요.

10. 자주 발생하는 문제

증상원인해결 방법
Cloud Run 배포 실패 Dockerfile 빌드 오류 또는 메모리 부족 Actions 로그 확인 → Docker 빌드 단계 오류 메시지 확인
DB 연결 실패 Secrets 값 오류 또는 Cloud SQL IP 방화벽 Cloud Run 로그 확인 → DB_HOST 값 검증
마이그레이션 컬럼 없음 오류 배포 전 마이그레이션 미실행 alembic upgrade head 실행 후 Cloud Run 재배포
CORS 에러 CORS_EXTRA_ORIGINS 미설정 Cloud Run 환경변수에 Vercel URL 추가 (https:// 포함)
Vercel 빌드 실패 TypeScript 타입 오류 또는 환경변수 누락 Vercel 대시보드 → 해당 배포 → Build Logs 확인
Cold Start 지연 Cloud Run 최소 인스턴스 0개 설정 min-instances=1 로 설정 (비용 증가 감수)
