SECTION 01

무신사 수집, 한눈에 보기

KREAM과 다르게 무신사는 서버에서 직접 API를 호출해서 수집한다. 확장앱은 쿠키 전달만 담당.

1

동시 요청 수 (concurrency)

Rate limit 대응으로 동시에 1개 요청만 보냄

200ms

요청 간 딜레이

각 요청 사이 0.2초 대기 (request_interval)

5단계

혜택가 계산

판매가 → 쿠폰 → 등급할인 → 적립금 → 선할인

9장

이미지 최대 수집

썸네일 + goodsImages 배열에서 중복 제거

KREAM과의 차이

KREAM은 확장앱이 탭을 열어서 DOM을 직접 긁어야 함 (큐 방식). 무신사는 서버가 API를 직접 호출해서 수집. 대신 무신사 로그인 쿠키가 반드시 필요하고, 그 쿠키는 확장앱이 자동으로 캡처해서 서버에 전송해줌.

SECTION 02

쿠키 인증 — 확장앱이 자동으로

무신사 API는 로그인 쿠키 없이는 가격/재고 정보를 안 줌. 쿠키를 어떻게 얻는지 보자.

1

사용자가 무신사에 로그인

브라우저에서 musinsa.com에 로그인만 하면 됨. 다른 건 아무것도 안 해도 됨.

2

확장앱이 쿠키 자동 캡처

chrome.webRequest.onBeforeSendHeaders로 *.musinsa.com 요청의 Cookie 헤더를 가로챔. 쿠키가 바뀔 때만 감지.

extension/background.js → webRequest 리스너
3

3초 debounce 후 서버 전송

쿠키가 바뀌면 3초 기다렸다가 POST /proxy/musinsa/set-cookie로 서버에 전송. 서버가 로그인 유효성 검증 후 DB에 저장.

POST /api/v1/samba/proxy/musinsa/set-cookie
4

서버가 수집할 때 쿠키 사용

수집 시 DB에서 쿠키를 꺼내서 API 요청 헤더에 포함. SambaSettings 테이블에 key="musinsa_cookie"로 저장됨.

proxy/musinsa.py → MusinsaClient
주의

쿠키가 없거나 만료되면 수집이 HTTP 400으로 실패함. 프론트에서 "무신사 로그인 필요" 알림을 표시함. 로그인 상태는 GET /proxy/musinsa/auth/status로 확인 가능.

SECTION 03

수집 흐름 — 서버가 직접 API 호출

키워드/URL/필터 3가지 방식으로 수집 가능. 모두 SSE 스트리밍으로 진행 상황을 실시간 전달.

1

프론트에서 수집 요청

사용자가 키워드/URL/필터를 입력하고 수집 시작. 프론트가 SSE 연결을 열고 대기.

POST /collector/collect-by-keyword
2

쿠키 검증

DB에서 무신사 쿠키 조회. 없으면 즉시 HTTP 400 반환. 있으면 MusinsaClient 생성.

3

검색 API 호출

GET api.musinsa.com/api2/dp/v1/plp/goods로 상품 목록 검색. 최대 200개씩 페이지네이션. 로그인 없이도 가능.

MusinsaClient.search_products()
4

상품별 상세 조회 (동시 1개)

각 상품에 대해 concurrency=1로 순차 호출. 200ms 딜레이 적용. 상세+옵션+재고+쿠폰+고시정보를 한번에 가져옴.

MusinsaClient.get_goods_detail()
5

DB 저장 + SSE 진행률 전송

수집된 상품을 batch insert. 각 상품 완료 시 SSE로 진행률을 JSON lines로 전달.

수집 방식 엔드포인트 설명
키워드 검색 /collector/collect-by-keyword "나이키 에어맥스" 같은 키워드로 검색 후 수집
URL 수집 /collector/collect-by-url 무신사 상품/검색 URL을 직접 입력해서 수집
필터 수집 /collector/collect-filter/{id} 저장된 필터 조건으로 일괄 수집

SECTION 04

5단계 혜택가 계산 — 핵심 로직

무신사는 할인이 겹겹이 쌓임. 최종 혜택가(bestBenefitPrice)까지의 계산 과정.

1

기본 판매가 (s_price)

immediateDiscountedPrice → salePrice → normalPrice 순서로 fallback. 즉시할인이 적용된 가격이 기본.

2

쿠폰 할인

goodsPrice.couponPrice가 0보다 크고 판매가보다 작으면 적용. benefit_coupon_discount = s_price - couponPrice

3

등급할인 (10원 절사)

int(benefit_base × memberDiscountRate / 100 / 10) × 10. 무신사 회원 등급에 따른 추가 할인.

4

적립금 사용 (10원 절사)

int(point_base × point_rate_pct / 100 / 10) × 10. 적립금으로 추가 차감.

5

적립 선할인 (isPrePoint=true만)

int(remaining × memberDiscountRate / 100 / 10) × 10. 최종 bestBenefitPrice = remaining - pre_discount

proxy/musinsa.py → get_goods_detail()
결과

수집 데이터에 originalPrice, salePrice, couponPrice, bestBenefitPrice, memberDiscountRate 다섯 가지 가격 필드가 저장됨. 정책 관리에서 마진 계산 시 bestBenefitPrice를 원가로 사용.

SECTION 05

Rate Limit 대응 — 차단 안 당하는 법

무신사는 요청이 빠르면 429/403으로 차단함. 세마포어 + 딜레이 + exponential backoff로 대응.

기본 전략

concurrency=1 (동시 1개) + 200ms 딜레이로 초당 최대 5개 요청

차단 감지

HTTP 429/403 → RateLimitError 예외 발생. Retry-After 헤더 확인.

Exponential Backoff

차단 시 인터벌 2배 증가 (0.4→0.8→1.6→...최대 30초). 성공 시 점진 복원.

긴급 중단

연속 5회 이상 차단 → 해당 소싱처 전체 일시 중단. 서비스 보호.

# refresher.py — 차단 대응 흐름

try:
    result = await MusinsaClient.get_goods_detail(goods_no)
    _site_intervals["MUSINSA"] = max(1.0, interval - 0.5)  # 성공 시 점진 복원
    _site_consecutive_errors["MUSINSA"] = 0

except RateLimitError as e:
    _site_intervals["MUSINSA"] *= 2                        # 인터벌 2배
    _site_intervals["MUSINSA"] = min(30, interval)        # 최대 30초
    _site_consecutive_errors["MUSINSA"] += 1

    if e.retry_after > 0:
        await asyncio.sleep(e.retry_after)                # Retry-After 대기

    if _site_consecutive_errors["MUSINSA"] >= 5:
        # 전체 수집 일시 중단

SECTION 06

옵션 & 재고 처리 — 판정 규칙

무신사 옵션/재고 API 응답을 어떻게 해석하는지.

조건 stock isSoldOut 설명
activated=false 또는 isDeleted=true - - 옵션 목록에서 제외
outOfStock=true + isRedirect=false 0 true 품절
isRedirect=true (브랜드직배) null false 재고 미확인 (판매 가능)
remainQuantity 존재 값 그대로 false 실재고 수량
그 외 999 false 기본값 (재고 충분 추정)
옵션명 규칙

옵션의 optionValues 배열에서 각 name을 " / "로 조인. 예: 블랙 / 270

SECTION 07

이미지 처리 — CDN 차단 대응

무신사 이미지는 Referer 체크가 있어서 단순 다운로드가 안 됨.

수집 순서

1. thumbnailImageUrl → images[0]
2. goodsImages[] 배열 → images[1~8]
3. 중복 제거 (dict.fromkeys)
4. 최대 9장 제한
5. goodsContents HTML에서 <img src> 파싱 (icon/btn_ 제외)

CDN 핫링크 차단

이미지 다운로드 시 Referer: https://www.musinsa.com/ 헤더 필수.
응답이 1000바이트 미만이면 차단 이미지(플레이스홀더)로 판단하고 스킵. 

# URL 변환 규칙 (_to_image_url)

path가 http로 시작   → 그대로 사용
path가 //로 시작     → https: 붙임
path가 /로 시작      → https://image.msscdn.net 붙임

SECTION 08

고시정보 추출 — 필드 매핑

마켓 등록에 필요한 고시정보를 무신사 API에서 자동 추출.

필드 매칭 조건 예시
소재 (material)"소재" 또는 "재질" 포함폴리에스터 100%
색상 (color)"색상" 정확 일치BLACK
사이즈 (size)"치수" 또는 "사이즈" 포함, "취급" 미포함FREE(44~110)
제조사 (manufacturer)"제조사" 또는 "제조자" 포함(주)나이키코리아
원산지 (origin)"제조국" 또는 "원산지" 포함베트남
세탁법 (careInstructions)"세탁" 또는 "취급" 포함, "치수" 미포함손세탁 권장
품질보증 (qualityGuarantee)"품질보증" 포함소비자분쟁해결기준
참고

갱신 모드(refresh_only=True)에서는 고시정보 API를 호출하지 않아 속도가 빨라짐. 최초 수집 시에만 가져옴.
