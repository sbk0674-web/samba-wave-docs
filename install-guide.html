📄 삼바웨이브 설치 가이드

Install Guide v1.0 2026-04-16

프로젝트 파일을 받은 후 로컬 개발 환경부터 프로덕션 레벨까지 단계별로 설정하는 완전 가이드입니다.

DB 마이그레이션 → 백엔드 서버 → 프론트엔드 → 확장앱

이 순서를 반드시 지킬 것. 역순으로 배포하면 없는 컬럼/API 참조로 서비스 장애가 발생합니다.

💾 1. 프로젝트 클론 & 구조 이해

// 저장소 클론
git clone <repo-url> samba-wave
cd samba-wave

디렉토리 구조

samba-wave/
├── backend/                    # Python FastAPI 백엔드
│   ├── backend/
│   │   ├── api/v1/routers/    # API 라우터 (도메인별)
│   │   ├── core/config.py     # 환경변수 설정 (Pydantic BaseSettings)
│   │   ├── db/orm.py          # Read/Write DB 세션 팩토리
│   │   ├── domain/samba/      # 핵심 비즈니스 로직 (DDD)
│   │   │   ├── collector/     # 상품 수집
│   │   │   ├── policy/        # 가격 정책
│   │   │   ├── shipment/      # 발송 처리
│   │   │   ├── order/         # 주문 관리
│   │   │   ├── proxy/         # 마켓 API 연동
│   │   │   └── plugins/       # 소싱사이트별 수집 플러그인
│   │   └── dtos/              # Request/Response DTO
│   ├── alembic/               # DB 마이그레이션
│   └── pyproject.toml
├── frontend/                   # Next.js 15 프론트엔드
│   ├── src/app/               # App Router 페이지
│   │   └── samba/             # Samba Wave 메인 기능
│   ├── src/components/
│   ├── src/lib/
│   │   ├── api.ts             # API 클라이언트
│   │   └── samba/api.ts       # Samba 도메인 API
│   └── src/hooks/
└── .github/workflows/
    └── deploy-cloudrun.yml    # CI/CD 파이프라인
핵심 원칙: 백엔드는 각 도메인 디렉토리마다 model.py / service.py / repository.py 3-레이어 구조입니다. 새 기능 추가 시 DTO → Repository → Service → Router 순서로 작성하세요.

🐍 2. 백엔드 로컬 환경 설정

// 가상환경 생성 및 패키지 설치
cd backend

# 가상환경 생성
uv venv

# 가상환경 활성화
# Windows:
.venv\Scripts\activate
# Mac/Linux:
source .venv/bin/activate

# 패키지 설치
uv pip install -e .
uv pip install -e .[dev]

# .env 파일 생성
cp .env.example .env
주의: .env 파일은 절대 Git에 커밋하지 마세요. .gitignore에 포함되어 있지만, 반드시 확인하세요.

.env 필수 설정값

키설명필수
write_db_host Write DB 호스트 (예: localhost) ●
write_db_port Write DB 포트 (기본: 5432) ●
write_db_name Write DB 이름 ●
write_db_user Write DB 사용자명 ●
write_db_password Write DB 비밀번호 ●
read_db_host Read DB 호스트 (로컬에서는 write와 동일) ●
read_db_port Read DB 포트 ●
read_db_name Read DB 이름 ●
read_db_user Read DB 사용자명 ●
read_db_password Read DB 비밀번호 ●
jwt_secret_key JWT 서명 키 (임의의 긴 문자열) ●
environment 실행 환경: development 또는 production ●
gemini_api_key Google Gemini API 키 (AI 이미지/태그)
r2_account_id Cloudflare R2 Account ID
r2_access_key_id Cloudflare R2 Access Key
r2_secret_access_key Cloudflare R2 Secret Key
r2_bucket_name R2 버킷 이름
로컬 개발 팁: environment=development로 설정하면 디버그 로그가 활성화됩니다. read_db_*는 로컬에서 write DB와 동일하게 설정해도 무방합니다.

🗃 3. DB 설정 옵션

로컬 개발 환경에서는 두 가지 방법 중 하나를 선택하세요.

🐊 옵션 A: 로컬 PostgreSQL 추천

Docker로 PostgreSQL을 로컬에 실행합니다. 프로덕션 데이터와 완전히 분리되어 안전합니다.

☁ 옵션 B: Cloud SQL 직접 연결 프로덕션

실제 프로덕션 DB에 연결합니다. 데이터 변경 위험이 있으니 주의하세요.

옵션 A: Docker로 로컬 PostgreSQL 실행

// Docker로 PostgreSQL 16 실행
docker run -d \
  --name samba-db \
  -e POSTGRES_USER=samba \
  -e POSTGRES_PASSWORD=password \
  -e POSTGRES_DB=sambawave \
  -p 5432:5432 \
  postgres:16

위 명령 실행 후 .env 파일 설정:

write_db_host=localhost
write_db_port=5432
write_db_name=sambawave
write_db_user=samba
write_db_password=password
read_db_host=localhost
read_db_port=5432
read_db_name=sambawave
read_db_user=samba
read_db_password=password

옵션 B: Cloud SQL Proxy 사용 (프로덕션 연결)

// Cloud SQL Proxy 설치 및 실행 (Mac/Linux)
# Cloud SQL Proxy 다운로드
curl -o cloud-sql-proxy https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.9.0/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy

# 프록시 실행 (GCP 프로젝트 ID, 리전, 인스턴스명으로 교체)
./cloud-sql-proxy --port 5432 <PROJECT_ID>:<REGION>:<INSTANCE_NAME>
주의: Cloud SQL에 직접 연결하는 경우, 실제 프로덕션 데이터가 존재합니다. 마이그레이션 테스트는 반드시 로컬 DB에서 먼저 검증하세요.

🔄 4. DB 마이그레이션

// DB 스키마 생성 (최초 설치 시)
cd backend

# 모든 마이그레이션 적용
alembic upgrade head
성공 확인: INFO [alembic.runtime.migration] Running upgrade ... -> ... 메시지가 출력되면 정상입니다.

마이그레이션 관련 추가 명령어

// 마이그레이션 파일 생성 (모델 변경 후)
# 변경 감지 후 마이그레이션 파일 자동 생성
alembic revision --autogenerate -m "변경 내용 설명"

# 생성된 파일 검토 후 적용
alembic upgrade head

# 이전 버전으로 롤백
alembic downgrade -1
DB 컬럼 추가 체크리스트: model.py 필드 추가 → alembic revision --autogenerate → 마이그레이션 파일 검토 → 로컬 alembic upgrade head 확인 → push (CI/CD가 프로덕션에 자동 적용)

🆕 5. 백엔드 서버 실행

// 개발 서버 실행 (권장)
cd backend
.venv\Scripts\python.exe run.py
// 또는 uvicorn 직접 실행
uvicorn backend.main:app --reload --port 28080
주의: uvicorn app.main:app 형태는 사용 금지입니다. 반드시 backend.main:app으로 실행하세요.

서버가 정상 실행되면 아래 URL에서 확인할 수 있습니다:

  • API 문서 (Swagger): http://localhost:28080/docs
  • ReDoc: http://localhost:28080/redoc

△ 6. 프론트엔드 설정

// 패키지 설치 및 환경변수 설정
cd frontend
pnpm install
cp .env.example .env.local

.env.local 필수 설정값

키설명필수
NEXT_PUBLIC_API_URL 백엔드 API URL (로컬: http://localhost:28080) ●
NEXTAUTH_SECRET NextAuth 서명 키 ●
NEXTAUTH_URL 로컬: http://localhost:3000 ●
// 개발 서버 실행
pnpm dev

실행 후 브라우저에서 http://localhost:3000/samba 접속

중요: 로컬 접속 주소는 항상 http://localhost:3000/samba입니다. http://localhost:3000으로 접속하면 리디렉션됩니다.

🐞 7. Chrome 확장앱 설치

  1. Chrome 브라우저에서 chrome://extensions 접속
  2. 우상단 개발자 모드 토글 ON
  3. 압축해제된 확장 프로그램 로드 클릭
  4. 프로젝트 루트의 extension/ 폴더 선택
  5. 확장앱 아이콘 클릭 후 백엔드 URL 설정: http://localhost:28080
디버깅 팁: Chrome을 --remote-debugging-port=9222 옵션으로 실행하면 확장앱 내부 상태를 디버깅할 수 있습니다.
설계 원칙: 확장앱은 얇은 클라이언트입니다. DOM 읽기와 서버 전송만 담당하며, 비즈니스 로직은 백엔드에 위치합니다.

🔐 8. 초기 설정 (최초 1회)

서비스를 처음 사용하기 전 설정 페이지(/samba/settings)에서 아래 항목을 순서대로 설정하세요.

  1. 관리자 계정 생성
    처음 가입한 계정이 자동으로 관리자 권한을 갖습니다.
  2. 마켓 계정 연결
    스마트스토어, 쿠팡, 11번가 등 판매처 API Key를 설정 페이지에서 입력합니다.
  3. 소싱처 쿠키 수집
    Chrome 확장앱으로 무신사, KREAM 등 소싱처에 로그인 후 쿠키를 전송합니다.
  4. Cloudflare R2 설정
    AI 배경제거 이미지가 저장될 R2 버킷 정보를 입력합니다 (Account ID, Access Key, Bucket Name).
  5. Gemini / Claude API 키 설정
    AI 이미지 처리(Gemini)와 카테고리 매핑(Claude)에 사용되는 API 키를 입력합니다.

▲ 9. 프로덕션 배포

상세 내용은 배포 가이드를 참조하세요. 핵심 단계만 요약합니다.

  1. GitHub Secrets 등록
    GCP_SA_KEY, DB 연결 정보, JWT_SECRET_KEY 등을 GitHub 저장소 Settings > Secrets에 등록합니다.
  2. Cloud SQL 인스턴스 생성
    GCP Console에서 PostgreSQL 16 Cloud SQL 인스턴스를 생성합니다. Read Replica 분리를 권장합니다.
  3. Cloud Run 서비스 생성
    백엔드 컨테이너를 Cloud Run에 배포합니다. 메모리 2Gi 이상 권장.
  4. Vercel 프론트엔드 연결
    GitHub 저장소를 Vercel에 연결하고 NEXT_PUBLIC_API_URL을 Cloud Run URL로 설정합니다.
  5. GitHub Actions 자동 배포 확인
    main 브랜치에 push하면 Actions가 자동으로 빌드·배포합니다. 완료까지 약 3~5분 소요.
배포 순서 필수 준수: DB 마이그레이션 → 백엔드 → 프론트엔드 순서를 반드시 지키세요. 역순 배포 시 서비스 장애가 발생합니다.

🔥 10. 문제 해결

✕ DB 연결 실패: connection refused
원인: PostgreSQL이 실행되지 않았거나 .env의 포트/호스트가 잘못됨
해결: docker ps로 컨테이너 상태 확인 → docker start samba-db 재실행. .env의 write_db_host/port 재확인
✕ alembic upgrade head 실패: relation does not exist
원인: 마이그레이션 파일 순서 충돌 또는 누락된 파일
해결: alembic history로 마이그레이션 히스토리 확인 → alembic stamp head로 상태 초기화 후 재시도. 또는 DB를 초기화 후 alembic upgrade head 재실행
✕ 확장앱 쿠키 전송 실패
원인: 백엔드 서버가 실행되지 않았거나 확장앱 URL 설정이 잘못됨
해결: 확장앱 팝업에서 백엔드 URL을 http://localhost:28080으로 재설정. chrome://extensions에서 확장앱 오류 로그 확인
✕ 마켓 API 인증 실패: 401 Unauthorized
원인: 마켓 API 키가 만료되었거나 설정이 잘못됨
해결: 설정 페이지에서 마켓 API Key를 재입력. 각 마켓 개발자 센터에서 키 유효성 확인. 쿠팡은 Vendor ID도 함께 확인 필요
✕ pnpm install 실패: engine incompatibility
원인: Node.js 버전이 요구사항(20+)보다 낮음
해결: node --version 확인 후 nvm 또는 공식 사이트에서 Node.js 20 LTS 설치
✕ uvicorn 실행 시 ModuleNotFoundError
원인: 가상환경이 활성화되지 않았거나 패키지가 설치되지 않음
해결: .venv\Scripts\activate (Windows) 또는 source .venv/bin/activate (Mac/Linux)로 가상환경 활성화 후 재시도
✕ ruff format 실패로 CI 빌드 실패
원인: Python 파일을 수정 후 ruff format 실행 없이 push함
해결: cd backend && .venv/Scripts/python.exe -m ruff format . 실행 후 다시 커밋·push. ruff check --fix .도 함께 실행할 것
더 많은 정보: 추가 문제는 개발 가이드 또는 팀 개발 가이드를 참조하세요.
